# 大唐三公六部系统

本技能通过**三公六部**架构实现多Agent协作，采用大唐贞观年间历史人物命名，
支持任意类型任务的多模态处理。

## 核心特性

- **任务通用性**：动态适配任意类型任务
- **多模态支持**：输入（文本/图片），输出（文档/图表/Word）
- **分层架构**：三公决策 → 六部执行
- **并行执行**：兵贵神速，动态并发控制
- **实时响应**：动态响应用户中途输入
- **进度监控**：整体进度、Agent健康度、排队状态
- **实时输出**：各Agent发言即时呈现给用户

## 工作流程

### Phase 1: 太师府规划

1. **接收圣旨**：分析用户任务意图（支持多模态输入）
2. **判断任务等级**：
   - 闲聊 → 直接回复
   - 简单 → 单Agent执行
   - 中等 → 串行工作流
   - 复杂 → 完整三公六部流程
   - **讨论议题** → 自由讨论流程（见下文）
3. **制定奏折**：输出 `workspace/zouzhang.json`

## 自由讨论流程

当太师判断为**讨论议题**时（如技术选型争议、方案对比、需要多方观点），启动自由讨论：

### 判断标准

```yaml
讨论议题特征:
  - 包含"讨论"、"争议"、"选择"、"对比"等关键词
  - 无明确执行目标，而是寻求建议/决策
  - 需要多方视角综合考量
  - 例如: "讨论下数据库选型"、"这个方案怎么样"
```

### Phase D1: 议题讨论阶段

**目标**：确保议题正确、完整，拆分子议题，确定讨论框架

```yaml
Step 1: 议题确认
  主持人宣布原始议题
  各大臣发言确认议题理解是否正确
  可提出议题修改或完善建议

Step 2: 议题拆分
  太师房玄龄主导拆分子议题
  确保每个子议题独立、完整
  输出议题树结构

Step 3: 讨论框架
  确定讨论维度（如：技术、成本、风险、收益）
  每个维度指定负责大臣
  确保讨论结果完备
```

**议题讨论输出**：

```markdown
## 【议题确认】

**原始议题**: [议题]

**子议题拆分**:
1. [子议题1]
2. [子议题2]
3. [子议题3]

**讨论维度**:
| 维度 | 负责大臣 | 关注点 |
|------|---------|--------|
| 成本效益 | 户部刘晏 | 投入产出、收益周期 |
| 技术实现 | 工部阎立德 | 架构、安全、可维护性 |
| 执行落地 | 兵部李靖 | 推广、效率、时间表 |
| 风险合规 | 刑部狄仁杰 | 法律、风险、隐私 |
| 战略价值 | 太师房玄龄 | 长远影响、战略意义 |
```

### Phase D2: 信息收集阶段

**目标**：各大臣补充资料，建立共享信息池

```yaml
信息来源:
  外网搜索: 百度搜索（实时网络信息）
  内网搜索: 本地文件搜索（已有文档资料）
  工作网络: 衔风搜索（企业内部知识库）

搜索流程:
  1. 大臣发言时可提出"需补充信息"
  2. 主持人批准后执行搜索
  3. 搜索结果写入共享池
  4. 标注来源，供所有大臣查阅

共享池格式:
  workspace/shared-pool/
  ├── key-findings.md      # 关键发现汇总
  ├── sources.json         # 来源索引
  └── details/             # 详细资料
      ├── baidu-xxx.md     # 百度搜索结果
      ├── local-xxx.md     # 本地文件
      └── feishu-xxx.md    # 衔风文档
```

### Phase D3.5: 补充信息处理阶段（新增）

**目标**：汇总大臣提出的信息需求，执行补充搜索或标记待研究

```yaml
执行时机: 第一轮发言结束后，自由补充阶段前

执行流程:
  Step 1: 汇总信息需求
    - 收集所有大臣提出的"需补充信息"
    - 分类整理（按信息类型：数据类、政策类、技术类等）
    - 标注优先级（高/中/低）

  Step 2: 判断可获取性
    可获取信息:
      - 外网公开信息 → 百度搜索获取
      - 本地已有资料 → 文件搜索获取
      - 企业知识库 → 衔风搜索获取
    
    不可获取信息:
      - 未公开的政策细节 → 标记"待官方披露"
      - 内部保密数据 → 标记"需权限获取"
      - 未来规划信息 → 标记"待进一步研究"

  Step 3: 执行补充搜索（可获取部分）
    - 兵部李靖执行搜索任务
    - 搜索结果追加到共享池
    - 更新 shared-pool/key-findings.md

  Step 4: 输出补充信息报告
    格式:
    ## 【补充信息处理报告】
    
    ### 已获取补充信息
    | 信息需求 | 提出者 | 获取方式 | 关键发现 |
    |---------|--------|---------|---------|
    | [需求1] | 户部 | 百度搜索 | [发现] |
    
    ### 待获取信息（标记）
    | 信息需求 | 提出者 | 原因 | 建议 |
    |---------|--------|------|------|
    | [需求2] | 刑部 | 未公开 | 待官方披露 |
    
    ---
    已添加到共享池，供后续发言参考。

  Step 5: 通知大臣查阅
    - 主持人宣布："补充信息已获取，各位可查阅共享池更新"
    - 自由补充阶段的大臣可引用新获取信息
```

### Phase D4: 自由补充阶段

**主持人**：太保长孙无忌

```yaml
前置条件: 补充信息已处理（Phase D3.5完成）

发言协调机制（举手-点名）:

  Step 1: 举手阶段
    - 主持人询问"谁要补充？"
    - 想发言的Agent返回"举手"信号
    - 主持人收集举手名单

  Step 2: 点名顺序
    - 反对方优先发言（阐述反对理由）
    - 然后支持方回应
    - 最后中立方可补充
    - 同优先级按原发言顺序

  Step 3: 串行发言
    - 主持人点名："[大臣名]，请发言"
    - 该大臣发言后，主持人点名下一位
    - 避免多人同时发言冲突
    
  Step 4: 引用补充信息
    - 大臣可引用Phase D3.5获取的新信息
    - 标注"来源: 补充搜索"
    - 如仍有信息缺失，可继续提出需补充信息
```

### Phase D5: 共识决议阶段

**目标**：汇总各方观点，形成最终决议

```yaml

  反对≥3人: 继续讨论或提交用户裁决
```

## 实时会议纪要机制

**目标**：全程记录讨论过程，确保用户能看到每个Agent的完整发言，形成可追溯的会议纪要

**【核心原则】Agent发言必须完整、原样呈现给用户，主持人汇总不能替代原始发言**

```yaml
发言呈现规则:
  每位Agent发言后:
    1. 【立即】完整输出Agent的原始发言内容给用户
    2. 【然后】主持人做简短过渡（可选，不超过20字）
    3. 【同时】追加完整发言原文到 speakers-log.md
    
  禁止行为:
    ❌ 主持人总结替代原始发言
    ❌ 精简/压缩Agent发言内容
    ❌ 只输出观点摘要而不展示论证过程
    ❌ 省略Agent引用的数据和论据
    
  正确示例:
    ✅ 先显示: "【户部刘晏发言】\n立场: 有条件支持\n观点: ...\n论据: 1. xxx 2. xxx"
    ✅ 后过渡: "主持人: 刘晏从成本角度发言完毕，下一位..."

会议纪要结构:
  workspace/meeting-minutes/
  └── {YYYYMMDD}_{议题关键词}/        # 【新增】按日期+议题归档
      ├── state.json                   # 状态追踪（实时更新）
      ├── agenda.md                    # 议题确认（Phase D1）
      ├── speakers-log.md              # 发言记录（实时追加，完整原文）
      ├── shared-pool.md               # 共享信息池汇总
      ├── info-requests.md             # 补充信息需求汇总（Phase D3.5）
      ├── supplement-report.md         # 补充信息处理报告（Phase D3.5）
      ├── decisions.md                 # 决议记录（Phase D5）
      ├── action-items.md              # 待办事项
      ├── conclusion.md                # 综合陈述（Phase D4）
      └── complete-summary.md          # 完整会议纪要（Phase D6生成）

目录命名规则:
  - 格式: {YYYYMMDD}_{议题关键词}
  - 示例: 20260408_数字人民币运营机构扩容
  - 示例: 20260409_数据库选型讨论
  - 议题关键词: 从议题中提取核心词，不超过20字

纪要更新时机:
  - 议题确认后 → 更新 agenda.md
  - 每位大臣发言后 → 【立即】追加完整发言原文到 speakers-log.md
  - 第一轮发言结束 → 汇总信息需求到 info-requests.md
  - 补充搜索完成 → 更新 supplement-report.md 和 shared-pool.md
  - 新信息加入共享池 → 更新 shared-pool.md
  - 形成决议后 → 更新 decisions.md
  - 有待办事项 → 更新 action-items.md
  - 讨论结束后 → 【必须】礼部生成 complete-summary.md（包含所有发言完整原文）
```

**会议纪要格式**：

```markdown
# 【朝堂议事纪要】

## 基本信息
- **议题**: [议题名称]
- **时间**: 2026-04-02
- **主持人**: 太保长孙无忌
- **参与大臣**: [列表]

## 一、议题确认

**原始议题**: [用户提出的议题]
```

### 工作流程

```yaml

  主持人宣布议题后:
    1. 兵部执行搜索任务（百度搜索、文件搜索）

Phase 2: 发言阶段
  所有Agent发言前:
    1. 先读取 shared-pool/key-findings.md
    2. 基于共享信息 + 个人视角发表观点

补充搜索:
  若某Agent发现需要更多信息:
    1. 提出补充搜索请求
    2. 主持人批准后，兵部执行补充搜索
    3. 新信息追加到共享池
    4. 通知所有Agent查阅更新
```

```yaml
  - 输出议题确认文档

  - 建立共享信息池

Step 6: Phase D3 - 第一轮发言
  - 各大臣按维度发言

  - 记录每位大臣提出的"需补充信息"

  - 汇总所有大臣的信息需求
  - 判断可获取性
  - 执行补充搜索（可获取部分）
```
