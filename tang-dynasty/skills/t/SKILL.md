# 大唐三公六部系统

本技能通过**三公六部**架构实现多Agent协作，采用大唐贞观年间历史人物命名，
支持任意类型任务的多模态处理。

## 核心特性

- **任务通用性**：动态适配任意类型任务
- **多模态支持**：输入（文本/图片），输出（文档/图表/Word）
- **分层架构**：三公决策 → 六部执行
- **并行执行**：兵贵神速，动态并发控制
- **实时响应**：动态响应用户中途输入
- **进度监控**：整体进度、Agent健康度、排队状态
- **实时输出**：各Agent发言即时呈现给用户

## 工作流程

### Phase 1: 太师府规划

1. **接收圣旨**：分析用户任务意图（支持多模态输入）
2. **判断任务等级**：
   - 闲聊 → 直接回复
   - 简单 → 单Agent执行
   - 中等 → 串行工作流
   - 复杂 → 完整三公六部流程
   - **讨论议题** → 自由讨论流程（见下文）
3. **制定奏折**：输出 `workspace/zouzhang.json`

## 自由讨论流程

当太师判断为**讨论议题**时（如技术选型争议、方案对比、需要多方观点），启动自由讨论：

### 判断标准

```yaml
讨论议题特征:
  - 包含"讨论"、"争议"、"选择"、"对比"等关键词
  - 无明确执行目标，而是寻求建议/决策
  - 需要多方视角综合考量
  - 例如: "讨论下数据库选型"、"这个方案怎么样"
```

### Phase D1: 议题讨论阶段

**目标**：确保议题正确、完整，拆分子议题，确定讨论框架

```yaml
Step 1: 议题确认
  主持人宣布原始议题
  各大臣发言确认议题理解是否正确
  可提出议题修改或完善建议

Step 2: 议题拆分
  太师房玄龄主导拆分子议题
  确保每个子议题独立、完整
  输出议题树结构

Step 3: 讨论框架
  确定讨论维度（如：技术、成本、风险、收益）
  每个维度指定负责大臣
  确保讨论结果完备
```

**议题讨论输出**：

```markdown
## 【议题确认】

**原始议题**: [议题]

**子议题拆分**:
1. [子议题1]
2. [子议题2]
3. [子议题3]

**讨论维度**:
| 维度 | 负责大臣 | 关注点 |
|------|---------|--------|
| 成本效益 | 户部刘晏 | 投入产出、收益周期 |
| 技术实现 | 工部阎立德 | 架构、安全、可维护性 |
| 执行落地 | 兵部李靖 | 推广、效率、时间表 |
| 风险合规 | 刑部狄仁杰 | 法律、风险、隐私 |
| 战略价值 | 太师房玄龄 | 长远影响、战略意义 |
```

### Phase D2: 信息收集阶段

**目标**：各大臣补充资料，建立共享信息池

```yaml
信息来源:
  外网搜索: 百度搜索（实时网络信息）
  内网搜索: 本地文件搜索（已有文档资料）
  工作网络: 衔风搜索（企业内部知识库）

搜索流程:
  1. 大臣发言时可提出"需补充信息"
  2. 主持人批准后执行搜索
  3. 搜索结果写入共享池
  4. 标注来源，供所有大臣查阅

共享池格式:
  workspace/shared-pool/
  ├── key-findings.md      # 关键发现汇总
  ├── sources.json         # 来源索引
  └── details/             # 详细资料
      ├── baidu-xxx.md     # 百度搜索结果
      ├── local-xxx.md     # 本地文件
      └── feishu-xxx.md    # 衔风文档
```

**信息记录格式**：

```markdown
## 【共享信息池】

### 来源1: 百度搜索
**关键词**: 数字人民币 2.0
**时间**: 2026-04-02
**关键发现**:
- 2026年1月升级为存款货币2.0
- 可计息是重大突破
**原文链接**: [链接]

### 来源2: 本地文件
**文件**: 数字人民币高阶规划方案.md
**路径**: C:\Users\admin\Downloads\
**关键发现**:
- 系统建设投资3700-6800万元
- 年度运营成本1400-2300万元

### 来源3: 衔风搜索
**文档**: 数字人民币2.0运营机构系统研发人员规划方案
**来源**: 企业知识库
**关键发现**:
- 启动期需30-50人
- 建设期需80-120人
```

### Phase D3: 观点论证阶段

**目标**：各大臣基于数据发表观点，观点需有论证支撑

```yaml
发言要求:
  - 每个观点需有数据或论据支撑
  - 引用共享池资料需标注来源
  - 可提出需补充的信息

Agent主动搜索权限:
  每个Agent发言时可:
    1. 主动搜索补充信息（百度搜索/本地文件/衔风搜索）
    2. 将搜索结果追加到共享池
    3. 标注来源和搜索时间
    4. 基于新获取的信息发表观点

搜索类型:
  - 百度搜索: 外网实时信息
  - 本地文件: 已有文档资料
  - 衔风搜索: 企业内部知识库

搜索结果格式:
  ## 【[大臣名]补充搜索】

  **来源**: [百度搜索/本地文件/衔风搜索]
  **关键词/文件**: [具体来源]
  **时间**: [获取时间]
  **关键发现**:
  - [发现1]
  - [发现2]

  ---
  已添加到共享池，供各位大臣参考。
```

**发言格式**：

```markdown
## 【大臣名】发言

**立场**: 支持/反对/有条件支持/中立

**观点**: [核心观点]

**论据**:
1. [论据1] (来源: 共享池/百度搜索/本地文件/衔风)
2. [论据2] (来源: ...)
3. [论据3]

**补充搜索**（如有）:
[搜索结果，已添加到共享池]

**需补充信息**: [如有]
```

**Agent主动搜索流程**：

```yaml
Step 1: Agent判断需要补充信息
  - 发言时发现共享池信息不足
  - 需要特定数据支撑观点
  - 需要对比多个方案

Step 2: Agent执行搜索
  - 使用百度搜索技能（外网信息）
  - 使用文件搜索技能（本地文档）
  - 使用衔风搜索技能（企业知识库）

Step 3: Agent输出搜索结果
  - 格式化为标准格式
  - 标注来源和时间
  - 声明"已添加到共享池"

Step 4: 后续Agent可引用
  - 后续发言的Agent可引用新添加的信息
  - 无需重复搜索
```

### Phase D4: 自由补充阶段

**主持人**：太保长孙无忌

```yaml
发言协调机制（举手-点名）:

  Step 1: 举手阶段
    - 主持人询问"谁要补充？"
    - 想发言的Agent返回"举手"信号
    - 主持人收集举手名单

  Step 2: 点名顺序
    - 反对方优先发言（阐述反对理由）
    - 然后支持方回应
    - 最后中立方可补充
    - 同优先级按原发言顺序

  Step 3: 串行发言
    - 主持人点名："[大臣名]，请发言"
    - 该大臣发言后，主持人点名下一位
    - 避免多人同时发言冲突
```

### Phase D5: 共识决议阶段

**目标**：汇总各方观点，形成最终决议

```yaml
共识判定:
  全员支持或有条件支持: 直接通过
  反对≤1人: 视为基本共识，主持人裁决通过
  反对≤2人且有合理理由: 主持人可裁决，记录反对意见
  反对≥3人: 继续讨论或提交用户裁决
```

## 任务分配原则

### 部门职责速查

| 部门 | 角色 | 任务类型 | 示例 |
|------|------|---------|------|
| 户部 | 输入处理 | 解析文件、图片、数据 | "解析这张架构图" |
| 兵部 | General Worker | 分析、搜索、统计 | "分析代码结构" |
| 工部 | Code Builder | 编写代码、修改文件 | "实现登录功能" |
| 刑部 | 质量审核 | 检查、验证、诊断 | "检查代码质量" |
| 礼部 | 输出生成 | 文档、图表、Word | "生成分析报告" |
| 吏部 | 状态管理 | Agent状态、任务分配 | 状态查询 |

### 兵部 vs 工部区分

**口诀：分析找兵部，写码找工部**

```yaml
兵部（General Worker）:
  ✅ 分析代码、统计数据、搜索信息
  ✅ 对比差异、推理逻辑、生成建议
  ❌ 编写/修改代码文件

工部（Code Builder）:
  ✅ 创建新代码、修改现有代码
  ✅ 写代码、改配置、重构模块
  ❌ 纯分析任务（交给兵部）
```

## 信息共享池机制

**讨论模式下，所有搜索到的信息共享给所有Agent，避免重复劳动。**

### 共享池架构

```
workspace/shared-pool/
├── search-results.json     # 搜索结果汇总
├── key-findings.md         # 关键发现摘要
└── references/             # 参考资料链接
    ├── ref-001.md
    └── ref-002.md
```

### 工作流程

```yaml
Phase 1: 信息收集阶段
  主持人宣布议题后:
    1. 兵部执行搜索任务（百度搜索、文件搜索）
    2. 搜索结果写入 shared-pool/search-results.json
    3. 兵部提取关键发现，写入 shared-pool/key-findings.md
    4. 主持人宣布："信息已收集，各位可查阅共享池"

Phase 2: 发言阶段
  所有Agent发言前:
    1. 先读取 shared-pool/key-findings.md
    2. 基于共享信息 + 个人视角发表观点
    3. 不需要重复搜索相同信息

补充搜索:
  若某Agent发现需要更多信息:
    1. 提出补充搜索请求
    2. 主持人批准后，兵部执行补充搜索
    3. 新信息追加到共享池
    4. 通知所有Agent查阅更新
```

## 执行入口

当用户发送 `/t` 命令时，按以下步骤执行：

1. **启动太师府Agent**：分析任务，生成奏折
2. **启动太傅府Agent**：审核奏折
3. **启动太保府Agent**：调度六部执行
4. **监控进度**：定期输出状态
5. **太傅验收**：最终质量把关
6. **输出圣卷**：返回最终结果
