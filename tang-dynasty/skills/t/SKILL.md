# 大唐三公六部系统

本技能通过**三公六部**架构实现多Agent协作，采用大唐贞观年间历史人物命名，
支持任意类型任务的多模态处理。

## 核心特性

- **任务通用性**：动态适配任意类型任务
- **多模态支持**：输入（文本/图片），输出（文档/图表/Word）
- **分层架构**：三公决策 → 六部执行
- **并行执行**：兵贵神速，动态并发控制
- **实时响应**：动态响应用户中途输入
- **进度监控**：整体进度、Agent健康度、排队状态
- **实时输出**：各Agent发言即时呈现给用户

## 工作流程

### Phase 1: 太师府规划

1. **接收圣旨**：分析用户任务意图（支持多模态输入）
2. **判断任务等级**：
   - 闲聊 → 直接回复
   - 简单 → 单Agent执行
   - 中等 → 串行工作流
   - 复杂 → 完整三公六部流程
   - **讨论议题** → 自由讨论流程（见下文）
3. **制定奏折**：输出 `workspace/zouzhang.json`

## 自由讨论流程

当太师判断为**讨论议题**时（如技术选型争议、方案对比、需要多方观点），启动自由讨论：

### 判断标准

```yaml
讨论议题特征:
  - 包含"讨论"、"争议"、"选择"、"对比"等关键词
  - 无明确执行目标，而是寻求建议/决策
  - 需要多方视角综合考量
  - 例如: "讨论下数据库选型"、"这个方案怎么样"
```

### Phase D1: 顺序发言

**主持人**：太保长孙无忌（协调各方是他的本职）

```yaml
流程:
  1. 主持人宣布议题
  2. 各Agent按顺序发言（观点+立场）
  3. 主持人汇总，统计分歧

发言顺序（按议题相关性）:
  - 与议题最相关的部门优先
  - 例如技术选型: 工部→兵部→刑部→太傅→其他
  - 例如资源分配: 户部→吏部→兵部→太师→其他
```

**发言格式**：

```markdown
## 【大臣名】发言

**立场**: 支持 / 反对 / 有条件支持 / 中立

**观点**: [核心观点一句话]

**论据**: [2-3条理由]

**建议**: [具体建议如有]
```

**【重要】实时输出要求**：

讨论模式下，每个Agent发言后，主控必须**立即输出**该Agent的发言内容给用户，而不是等所有Agent发完再汇总。

```yaml
执行流程:
  1. 调用Agent发言
  2. 收到Agent回复后，立即输出发言内容
  3. 再调用下一个Agent
  4. 重复直到所有Agent发言完毕
  5. 主持人汇总

错误做法:
  ❌ 并行调用所有Agent，最后统一输出
  ❌ 只输出汇总结果，不输出各Agent发言

正确做法:
  ✅ 串行调用，每收到一个回复就输出
  ✅ 用户可以实时看到每个大臣的观点
```

### Phase D2: 主持人汇总

每轮发言后，主持人输出：

```markdown
## 【主持人汇总】第N轮

### 立场统计
| 立场 | 人数 | 大臣 |
|------|------|------|
| 支持 | 3 | 工部、兵部、礼部 |
| 反对 | 2 | 太傅、刑部 |
| 有条件支持 | 1 | 户部 |
| 中立 | 2 | 吏部、太师 |

### 共识度: 62.5%（5/8支持或有条件支持）

### 争议焦点
1. [焦点描述]
2. [焦点描述]

### 主要观点
- 支持方: [核心论点]
- 反对方: [核心论点]
```

### Phase D3: 自由补充

主持人询问："是否有大臣需要补充或回应？"

```yaml
发言协调机制（举手-点名）:

  Step 1: 举手阶段
    - 主持人询问"谁要补充？"
    - 想发言的Agent返回"举手"信号
    - 主持人收集举手名单

  Step 2: 点名顺序
    - 反对方优先发言（阐述反对理由）
    - 然后支持方回应
    - 最后中立方可补充
    - 同优先级按原发言顺序

  Step 3: 串行发言
    - 主持人点名："[大臣名]，请发言"
    - 该大臣发言后，主持人点名下一位
    - 避免多人同时发言冲突

示例流程:
  主持人: "魏征、狄仁杰举手反对，请魏征先发言"
  魏征发言...
  主持人: "请狄仁杰发言"
  狄仁杰发言...
  主持人: "李靖举手回应，请发言"
  李靖发言...

自由发言规则:
  - 每人补充不超过2次
  - 补充后立场可改变
  - 必须等主持人点名才能发言

触发继续讨论条件:
  - 反对≥3人 → 继续讨论
  - 反对意见有新论据 → 继续讨论
  - 有大臣明确要求补充 → 继续讨论
```

### Phase D4: 达成共识

```yaml
共识判定:
  全员支持或有条件支持: 直接通过
  反对≤1人: 视为基本共识，主持人裁决通过
  反对≤2人且有合理理由: 主持人可裁决，记录反对意见
  反对≥3人: 继续讨论或提交用户裁决
```

**共识输出**：

```markdown
## 【共识决议】

经[N]轮讨论，达成共识：

### 最终决议
[决议内容/方案]

### 各方最终立场
| 大臣 | 立场 | 核心观点 |
|------|------|---------|
| 工部 | 支持 | 技术可行 |
| 太傅 | 有条件支持 | 需加强测试 |
| ... | ... | ... |

### 待办事项（如有条件支持）
- [条件落实事项]

---
决议生效，可进入执行阶段。
```

**未达成共识**：

```markdown
## 【提交裁决】

经[N]轮讨论，分歧仍存：

### 方案对比
| 方案 | 支持者 | 反对者 | 核心争议 |
|------|--------|--------|---------|
| A方案 | 工部、兵部 | 太傅、刑部 | 安全性 |
| B方案 | 太傅、刑部 | 工部 | 性能 |

### 请陛下裁决
1. 采用方案A
2. 采用方案B
3. 采用折中方案C
4. 提供更多信息后继续讨论
```

### Phase 2: 太傅府审核

1. **审核奏折**：检查规划可行性
2. **批驳或通过**：不通过则返回太师重新规划（最多3次迭代）
3. **移交太保**：审核通过后进入执行阶段

### Phase 3: 太保府调度

1. **解析奏折**：读取任务分解和依赖关系
2. **查询吏部**：获取空闲Agent状态
3. **调度六部**：按依赖关系并行启动执行Agent
4. **监控进度**：定期输出进度报告，处理异常

### Phase 4: 六部执行

根据任务类型调度对应部门：
- **户部**：输入解析（图片、文件、数据）
- **兵部**：分析、搜索、统计（General Worker）
- **工部**：代码编写、工程构建（Code Builder）
- **刑部**：质量检查、问题诊断
- **礼部**：文档生成、图表绘制
- **吏部**：状态记录、Agent池管理

### Phase 5: 太傅府验收

1. **验收成果**：检查各部输出质量
2. **整改或通过**：不通过则要求重做
3. **汇总圣卷**：生成最终输出

## 任务分配原则

### 部门职责速查

| 部门 | 角色 | 任务类型 | 示例 |
|------|------|---------|------|
| 户部 | 输入处理 | 解析文件、图片、数据 | "解析这张架构图" |
| 兵部 | General Worker | 分析、搜索、统计 | "分析代码结构" |
| 工部 | Code Builder | 编写代码、修改文件 | "实现登录功能" |
| 刑部 | 质量审核 | 检查、验证、诊断 | "检查代码质量" |
| 礼部 | 输出生成 | 文档、图表、Word | "生成分析报告" |
| 吏部 | 状态管理 | Agent状态、任务分配 | 状态查询 |

### 兵部 vs 工部区分

**口诀：分析找兵部，写码找工部**

```yaml
兵部（General Worker）:
  ✅ 分析代码、统计数据、搜索信息
  ✅ 对比差异、推理逻辑、生成建议
  ❌ 编写/修改代码文件

工部（Code Builder）:
  ✅ 创建新代码、修改现有代码
  ✅ 写代码、改配置、重构模块
  ❌ 纯分析任务（交给兵部）
```

## 信息共享池机制

**讨论模式下，所有搜索到的信息共享给所有Agent，避免重复劳动。**

### 共享池架构

```
workspace/shared-pool/
├── search-results.json     # 搜索结果汇总
├── key-findings.md         # 关键发现摘要
└── references/             # 参考资料链接
    ├── ref-001.md
    └── ref-002.md
```

### 工作流程

```yaml
Phase 1: 信息收集阶段
  主持人宣布议题后:
    1. 兵部执行搜索任务（百度搜索、文件搜索）
    2. 搜索结果写入 shared-pool/search-results.json
    3. 兵部提取关键发现，写入 shared-pool/key-findings.md
    4. 主持人宣布："信息已收集，各位可查阅共享池"

Phase 2: 发言阶段
  所有Agent发言前:
    1. 先读取 shared-pool/key-findings.md
    2. 基于共享信息 + 个人视角发表观点
    3. 不需要重复搜索相同信息

补充搜索:
  若某Agent发现需要更多信息:
    1. 提出补充搜索请求
    2. 主持人批准后，兵部执行补充搜索
    3. 新信息追加到共享池
    4. 通知所有Agent查阅更新
```

### 共享池格式

**search-results.json**：

```json
{
  "topic": "议题名称",
  "collected_at": "2026-04-02T10:00:00",
  "searches": [
    {
      "source": "百度搜索",
      "keywords": ["关键词1", "关键词2"],
      "results": [
        {"title": "...", "url": "...", "summary": "..."}
      ]
    },
    {
      "source": "文件搜索",
      "keywords": ["关键词"],
      "results": [
        {"file": "...", "matches": [...]}
      ]
    }
  ]
}
```

**key-findings.md**：

```markdown
# 关键发现汇总

## 来源
- 百度搜索: 150条结果，筛选35条高质量
- 文件搜索: 本地文档匹配结果

## 核心发现

### 发现1
[摘要内容]
来源: [来源链接]

### 发现2
[摘要内容]
来源: [来源链接]

## 数据要点
[关键数据汇总]

## 待补充信息
[需要进一步搜索的内容]
```

### Agent读取共享池

每个Agent发言前，主持人提示：

```markdown
## 【共享池更新】

信息已收集完毕，各大臣发言前请先查阅：

- 关键发现: shared-pool/key-findings.md
- 搜索结果: shared-pool/search-results.json

---
```

## 执行入口

当用户发送 `/t` 命令时，按以下步骤执行：

1. **启动太师府Agent**：分析任务，生成奏折
2. **启动太傅府Agent**：审核奏折
3. **启动太保府Agent**：调度六部执行
4. **监控进度**：定期输出状态
5. **太傅验收**：最终质量把关
6. **输出圣卷**：返回最终结果
