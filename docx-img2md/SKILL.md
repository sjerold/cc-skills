# Docx图片转Markdown 技能

将包含图片（截图/扫描件）的 docx 文档转换为 Markdown 格式。

## 使用场景

- docx 文档内容全部是截图或扫描件
- 需要将图片中的文字提取出来转为可编辑的 Markdown
- 流程图等图形类图片需要保留图片引用

## 操作步骤

### 1. 提取图片

用 Python 脚本从 docx 中提取所有嵌入图片到 `文字版/<文档名>/pic/` 目录：

```python
import sys
sys.path.insert(0, r'C:\Users\admin\.claude\skills\docx-img2md')
from extract_images import extract_images
extract_images('<docx文件路径>')
```

### 2. 逐张识别图片写入 md

**⚠️ 关键约束：禁止使用 Team 模式**

**必须使用独立的 `Agent()` 调用处理每张图片**，绝对不要创建 team 来协调图片识别任务。原因：team 会累积所有成员的完整配置和 inbox 消息，处理大量图片时请求体会超过 32MB 限制导致 session 崩溃。

**正确做法：每张处理完立即写入 md**

对 `文字版/<文档名>/pic/` 目录下的图片，使用独立的 `Agent()` 调用逐一处理。推荐每次并行启动 3-5 个 Agent，等它们全部完成后，立即用 `Edit` 追加结果到 md 文件，再处理下一批：

```
# 获取图片列表，每次处理 3-5 张
images = sorted(pic目录.glob("*.png"))

# 批次内并行，批次间串行，每批完成后立即写入 md
batch_size = 5
for i in range(0, len(images), batch_size):
    batch = images[i:i+batch_size]
    # 对 batch 中的每张图并行启动 Agent，在同一消息中发出所有 Agent 调用
    # 等所有 Agent 返回后，立即用 Edit 追加结果到 md 文件
    # 不要攒批，不要跳过写入步骤
```

**子 Agent 行为保证：**
- 每个子 Agent 只 `Read` **一张**图片，其上下文中只有一张图片的 base64 数据
- 子 Agent 先判断图片类型：
  - **文字类**：OCR 识别图片中的文字，按 Markdown 格式返回
  - **流程图/图形类**：返回 `![image](pic/xxx.png)` 保留原图引用，不做 OCR
- 子 Agent 完成后自动返回结果，base64 数据随子 Agent 上下文一起销毁
- 主 session 只保留子 Agent 返回的简短结果（几百字节）

**绝对不能做的：**
- ❌ 不要创建 Team 来协调图片识别
- ❌ 不要在主 session 中用 `Read` 直接读取图片（base64 会累积到主上下文）
- ❌ 不要让一个子 Agent 处理多张图片
- ❌ 不要把之前处理过的图片或文字重新传入子 Agent 上下文
- ❌ **禁止自己写 Python 脚本/调用 OCR 库批量处理图片**——必须用 `Agent()` 调用，让子 Agent 用视觉能力识别
- ❌ **禁止在主 session 中直接用视觉模型看图片**——base64 会进入主上下文导致膨胀
- ❌ **禁止跳过 Agent 调用直接用模型能力**——每张图片必须走 Agent() 流程

**强制执行规则（无论上下文多大都必须遵守）：**
1. 图片识别 **只能** 通过 `Agent()` 调用完成
2. 每个子 Agent **只读一张** 图片，完成后立即返回
3. 子 Agent 返回结果后 **立即** 用 `Edit` 追加到 md，不要攒批
4. 主 session 上下文膨胀时 **不要改变流程**，继续按上述规则处理下一批

**写入 md 文件时：**
- md 文件路径：`<docx同目录>/文字版/<文档名>/<文档名>.md`
- 用 `Write` 工具首次写入标题，后续用 `Edit` 工具追加内容
- 图片引用路径相对于 md 文件为 `pic/xxx.png`
- **区分两类图片**：
  - **文字类**（截图/扫描件/文档正文）：子 Agent 做 OCR，识别文字写入 md
  - **流程图/图形类**（见下方判断标准）：**不要 OCR**，直接保留原图引用 `![image](pic/xxx.png)`

### 3. 跨页表格处理（重要）

**问题**：同一个表格跨越多张图片时，每张图片只识别自己那部分，会导致：
- 表格被拆成多个独立表格块
- 表格行顺序错乱
- 表头重复出现

**处理方法**：
1. 识别到图片包含表格时，判断是否为跨页表格：
   - 图片底部表格没有完整的底边框线（表格被截断）
   - 图片顶部表格没有完整的顶边框线（表格从上一页延续）
   - 页眉/页脚出现在表格中间位置
   - 表格列数、列宽与前后图片一致

2. **如果确认是跨页表格**：
   - 第一页（有表头）：正常写入完整表格（表头+数据行）
   - 中间页（无表头，只有数据行）：将识别的数据行追加到上一个表格的末尾，**不要重复表头**
   - 最后一页（表格结束）：同样追加数据行到表格末尾

3. **Markdown 表格追加行的方法**：
   - 用 `Edit` 工具找到 md 文件中上一个表格的最后一行（`|---|---|---|` 分隔行下面的最后一行数据）
   - 在该行后面追加新的数据行
   - 确保列数一致

4. **完成所有图片后**：
   - 通读 md 文件，检查是否有断裂的表格
   - 将断裂的表格合并为一个完整表格

### 4. 完成

确认 md 文件包含所有图片的识别结果，图片引用路径正确，表格完整无断裂。

## 输出结构

```
<docx目录>/
├── 文档名.docx              # 原始文件
└── 文字版/                  # 输出目录
    ── <文档名>/            # 每个docx独立目录
        ├── pic/             # 该docx的图片
        │   ├── image_001.png
        │   ├── image_002.png
        │   ── ...
        └── <文档名>.md       # 转换结果
```

## 图片命名规则

格式：`image_001.png`

- 使用英文前缀 `image` + 三位序号（从 001 开始）
- 统一转为 PNG 格式
- **禁止使用中文或特殊字符作为文件名**，避免 Markdown 导入失败

## 流程图判断标准

如果图片包含以下特征，视为流程图/图形类：
- 箭头连接线、方框流程图
- 树状图、架构图、时序图
- UML 图、状态机图
- 饼图、柱状图等数据可视化图
- 包含 "如下图"、"流程图"、"示意图" 等文字

纯表格、段落文字的图片视为文字类。

## Markdown 换行规范（重要）

写入 md 文件时，必须遵守以下换行规则：

1. **列表项换行**：
   - a)、b)、c)... 或 1)、2)、3)... 等列表项，每项末尾加 `<br>` 强制换行
   - 错误：每项用普通换行（预览器可能合并为一行）
   - 正确：`a) 内容；<br>` `b) 内容；<br>` `c) 内容；`

2. **段落换行**：
   - 普通段落之间用一个空行分隔（两个换行符）
   - 同一段落内的换行用 `<br>` 标签

3. **表格单元格内的换行**：
   - 用 `<br>` 标签，不要用普通换行

4. **标题和正文**：
   - 标题行和正文之间空一行

5. **表格格式**：
   - 表头行：`| 列1 | 列2 | 列3 |`
   - 分隔行：`|------|------|------|`
   - 数据行：每行独立，`| 值1 | 值2 | 值3 |`
