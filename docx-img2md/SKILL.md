# Docx图片转Markdown 技能

将包含图片（截图/扫描件）的 docx 文档转换为 Markdown 格式。

## 使用场景

- docx 文档内容全部是截图或扫描件
- 需要将图片中的文字提取出来转为可编辑的 Markdown
- 流程图等图形类图片需要保留图片引用

## 操作步骤

### 1. 提取图片

用 Python 脚本从 docx 中提取所有嵌入图片到 `文字版/<文档名>/pic/` 目录：

**完整命令（必须使用 conda 的 dsbot_env 环境）：**

```bash
# Windows PowerShell 环境
cmd /c "call conda activate dsbot_env && python C:\Users\admin\.claude\plugins\docx-img2md\skills\docx-img2md\extract_images.py <docx文件路径>"
```

**️ 绝对禁止：**
- ❌ 不允许自己编写任何 Python 脚本来提取图片
- ❌ 不允许使用其他 Python 环境或依赖
- ❌ 不允许修改 extract_images.py 脚本
- ❌ 必须使用上述完整命令，包含完整的 conda 环境激活路径

### 2. 逐张识别图片写入 md

**⚠️ 关键约束：禁止使用 Team 模式**

**必须使用独立的 `Agent()` 调用处理每张图片**，绝对不要创建 team 来协调图片识别任务。原因：team 会累积所有成员的完整配置和 inbox 消息，处理大量图片时请求体会超过 32MB 限制导致 session 崩溃。

**正确做法：严格串行单线程处理**

对 `文字版/<文档名>/pic/` 目录下的图片，**严格按文件名排序，逐张串行处理**，绝对不要并行、不要分批：

```
# 获取图片列表，按文件名排序
images = sorted(pic目录.glob("*.png"))

# 严格按顺序逐张处理，一张完成后再处理下一张
for img in images:
    # 0. 【必须】运行进度检查脚本（基于文件状态，不依赖模型记忆）
    cmd /c "call conda activate dsbot_env && python C:\Users\admin\.claude\plugins\docx-img2md\skills\docx-img2md\check_progress.py <md文件路径>"
    # 如果输出 SELF_CHECK: required，必须先执行自检
    # 如果输出 SELF_CHECK: skip，继续处理

    # 1. 用 Agent() 读取并识别这一张图片
    # 2. 等 Agent 返回后，立即用 Edit 追加结果到 md
    # 3. 再处理下一张
    # 绝对不要并行启动多个 Agent
```

**进度检查脚本说明**：
- `check_progress.py` 统计 md 文件中已有的图片引用数量，基于**文件系统状态**判断是否需要自检
- 返回 `SELF_CHECK: required` 时 → 必须输出自检清单并确认通过
- 返回 `SELF_CHECK: skip` 时 → 输出下一张自检的触发条件，继续处理
- **绝对禁止跳过此步骤或用模型自行计数代替**

**子 Agent 行为保证：**
- 每次只启动 **一个** Agent，处理 **一张** 图片
- 子 Agent 只 `Read` 一张图片，其上下文中只有一张图片的 base64 数据
- 子 Agent 先判断图片类型（见下方"图片分类规则"）
- 子 Agent 完成后自动返回结果，base64 数据随子 Agent 上下文一起销毁
- 主 session 只保留子 Agent 返回的简短结果（几百字节）

**绝对不能做的：**
- ❌ 不要创建 Team 来协调图片识别
- ❌ 不要在主 session 中用 `Read` 直接读取图片（base64 会累积到主上下文）
- ❌ 不要让一个子 Agent 处理多张图片
- ❌ 不要把之前处理过的图片或文字重新传入子 Agent 上下文
- ❌ **禁止并行启动多个 Agent**——必须串行，一张处理完再处理下一张
- ❌ **禁止自己写 Python 脚本/调用 OCR 库批量处理图片**——必须用 `Agent()` 调用，让子 Agent 用视觉能力识别
- ❌ **禁止在主 session 中直接用视觉模型看图片**——base64 会进入主上下文导致膨胀
- ❌ **禁止跳过 Agent 调用直接用模型能力**——每张图片必须走 Agent() 流程

**强制执行规则（无论上下文多大都必须遵守）：**
1. 图片识别 **只能** 通过 `Agent()` 调用完成
2. **严格单线程串行**：每次只启动一个 Agent，处理一张图片
3. 子 Agent 返回结果后 **立即** 用 `Edit` 追加到 md
4. 按文件名排序顺序处理，**绝不打乱顺序**
5. 主 session 上下文膨胀时 **不要改变流程**，继续按上述规则处理下一张

**写入 md 文件时：**
- md 文件路径：`<docx同目录>/文字版/<文档名>/<文档名>.md`
- 用 `Write` 工具首次写入标题，后续用 `Edit` 工具追加内容
- 图片引用路径相对于 md 文件为 `pic/xxx.png`
- **图片分三类，处理方式不同**：
  - **纯文字类**（截图/扫描件/文档正文）：OCR 识别文字写入 md
  - **混合类**（既有文字又有流程图/图表）：OCR 识别文字写入 md，**同时保留原图引用** `![image](pic/xxx.png)` 插入文字下方
  - **纯图形类**（纯流程图/架构图/数据图表，无实质文字）：只保留原图引用 `![image](pic/xxx.png)`，不做 OCR

### 3. 跨页表格处理（重要）

**问题**：同一个表格跨越多张图片时，每张图片只识别自己那部分，会导致：
- 表格被拆成多个独立表格块
- 表格行顺序错乱
- 表头重复出现

**处理方法**：
1. 识别到图片包含表格时，判断是否为跨页表格：
   - 图片底部表格没有完整的底边框线（表格被截断）
   - 图片顶部表格没有完整的顶边框线（表格从上一页延续）
   - 页眉/页脚出现在表格中间位置
   - 表格列数、列宽与前后图片一致

2. **如果确认是跨页表格**：
   - 第一页（有表头）：正常写入完整表格（表头+数据行）
   - 中间页（无表头，只有数据行）：将识别的数据行追加到上一个表格的末尾，**不要重复表头**
   - 最后一页（表格结束）：同样追加数据行到表格末尾

3. **Markdown 表格追加行的方法**：
   - 用 `Edit` 工具找到 md 文件中上一个表格的最后一行（`|---|---|---|` 分隔行下面的最后一行数据）
   - 在该行后面追加新的数据行
   - 确保列数一致

4. **完成所有图片后**：
   - 通读 md 文件，检查是否有断裂的表格
   - 将断裂的表格合并为一个完整表格

### 4. 完成

确认 md 文件包含所有图片的识别结果，图片引用路径正确，表格完整无断裂。

## 输出结构

```
<docx目录>/
├── 文档名.docx              # 原始文件
└── 文字版/                  # 输出目录
    ── <文档名>/            # 每个docx独立目录
        ├── pic/             # 该docx的图片
        │   ├── image_001.png
        │   ├── image_002.png
        ── ...
        └── <文档名>.md       # 转换结果
```

## 图片命名规则

格式：`image_001.png`

- 使用英文前缀 `image` + 三位序号（从 001 开始）
- 统一转为 PNG 格式
- **禁止使用中文或特殊字符作为文件名**，避免 Markdown 导入失败

## 图片分类规则（重要）

### 第一步：初步判断

如果图片包含以下特征，**初步标记**为含图形元素：
- 箭头连接线、方框流程图
- 树状图、架构图、时序图
- UML 图、状态机图
- 饼图、柱状图等数据可视化图

纯表格、段落文字的图片初步判断为文字类。

### 第二步：二次确认（必须）

**初步标记为含图形的图片，必须进行二次确认**，因为很多以文字为主的图片会被误判。

二次确认标准：
- 图片中**图形元素占比超过 50%** → 确认为含图形
- 图片以**文字/表格为主，只有少量图标或装饰性图形** → 重新归类为文字类
- **带少量标注的表格** → 归类为文字类
- **带少量图标的纯文本段落** → 归类为文字类

### 第三步：根据分类处理

| 分类 | 处理方式 |
|------|---------|
| 纯文字类 | OCR 识别文字，写入 md |
| 混合类（文字+图形） | 用 `crop_graphics.py` 裁剪图形区域 + OCR 文字标注 + 裁剪图引用 |
| 纯图形类 | 仅保留原图引用 `![image](pic/xxx.png)` |

### 混合类图片的处理流程

混合类图片（既有文字又有流程图/图表）使用 `crop_graphics.py` 脚本处理：

**完整命令（必须使用 conda 的 dsbot_env 环境）：**

```bash
# Windows PowerShell 环境
cmd /c "call conda activate dsbot_env && python C:\Users\admin\.claude\plugins\docx-img2md\skills\docx-img2md\crop_graphics.py <图片路径> 文字版/<文档名>/pic/"
```

**️ 绝对禁止：**
- ❌ 不允许自己编写任何 Python 脚本来处理图片
- ❌ 不允许使用其他 Python 环境或依赖
- ❌ 不允许修改 crop_graphics.py 脚本
- ❌ 不允许直接调用 OCR 库或 OpenCV 自行处理
- ❌ 必须使用上述完整命令，包含完整的 conda 环境激活路径

**脚本功能**：
1. PaddleOCR 检测所有文字区域及边界框
2. 自动跳过页码、页脚、重复版本号（V01xxxxx_20251124）
3. 识别图注文字（"图1 xxx"）确定图形下边界
4. OpenCV 检测图形元素（流程图框、连线、形状）
5. 分析布局类型（纯文字/纯图形/混合）
6. 混合类：裁剪出图形区域，输出标注 JSON

**跳过的文字模式**：
- 纯数字（页码）：`1`, `12`, `- 3 -`
- 版本号：`V01B00E01_20251124`, `V01xxxxx_20251124`
- 分隔线：`---`, `———`
- 罗马数字页码：`I`, `II`, `III`
- 日期格式：`2025-11-24`, `2025/11/24`

**输出文件**：
- `xxx_graphic.png` — 裁剪后的纯图形部分（去除文字区域）
- `xxx_annotated.json` — 文字区域标注（含坐标、文字内容）

**md 写入格式**：
```
[OCR 识别的文字内容，按坐标排序]

![流程图](pic/xxx_graphic.png)
```

文字内容按从上到下、从左到右的顺序排列，图形引用放在最后。

## Markdown 换行规范（重要）

写入 md 文件时，必须遵守以下换行规则：

1. **列表项换行**：
   - a)、b)、c)... 或 1)、2)、3)... 等列表项，每项末尾加 `<br>` 强制换行
   - 错误：每项用普通换行（预览器可能合并为一行）
   - 正确：`a) 内容；<br>` `b) 内容；<br>` `c) 内容；`

2. **段落换行**：
   - 普通段落之间用一个空行分隔（两个换行符）
   - 同一段落内的换行用 `<br>` 标签

3. **表格单元格内的换行**：
   - 用 `<br>` 标签，不要用普通换行

4. **标题和正文**：
   - 标题行和正文之间空一行

5. **表格格式**：
   - 表头行：`| 列1 | 列2 | 列3 |`
   - 分隔行：`|------|------|------|`
   - 数据行：每行独立，`| 值1 | 值2 | 值3 |`
